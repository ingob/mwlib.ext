ReportLab Page Template

This is the 'master' for all of our pages. A script runs through the site taking the 'content' region of each other page, and placing it within a layout specified by this page.

How the site works...

We use a couple of Python scripts to help maintain the look and feel of the site. These are:
  refreshsite.py which controls everything
  faqmaker.py, written by Aaron and modified by Robin, which can convert marked-up text to both PDF and HTML.

Every page is modified in place :-( by a preprocessor.

Adding new pages

When adding a new page, or modifying an existing one, you don't need a full HTML page. Your page needs to contain two sections. One is a <title>My Page Title</title> section, which will be preserved and used in the finished page. The other is a 'content' section which goes between two structured comments:
  <!--User Content Starts Here-->   <!--User Content Ends Here-->

You then need to edit refreshsite.py and add the name of the file to the list of pages it keeps an eye on, then run it. The refreshsite.py script will preserve the title and 'Content' of your page, and put them into the right places in this template.

We did it this way as it seems to work for fancy HTML editors as well as for people using vi over a dialup connection.

Updating existing pages

When modifying a page later on, just edit freely between these two comments, and re-run refreshsite.py.

Updating the global template

Create a new template file in suitably vile colours, or with your added links; change the constant in refreshsite.py to point to it, and re-run refreshsite.py.

Simultaneous HTML and PDF files

Aaron provided a neat script to do this. You use an nroff-style markup in a text file which is easy to maintain; his script turns this into PDF and HTML. refreshsite.py has a list of such files which you should add to if adding more dual-format files: 
	# list of text files to output in HTML and PDF
	DUAL_FORMAT_FILES = [
		'devfaq.txt',
		'whitepaper.txt'
		]
It will recreate both HTML and PDF versions of these when run; so you should edit the text file versions of these files. the source of the current ones is here:
  devfaq.txt   whitepaper.txt

For now, we should work directly on the site; if you take a local copy, you should get it back up there before the people on the other side of the Atlantic start work, or let them know to avoid a particular file.
